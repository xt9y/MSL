# msl — macOS Subsystem for Linux

Build plan for a lightweight CLI tool using Apple's Virtualization.framework
that makes Arch Linux ARM feel like a native shell in Terminal.app.

---

## Project Structure

```
msl/
├── CMakeLists.txt
├── Resources/
│   └── msl.entitlements
├── scripts/
│   └── build-image.sh       # Build arch.img + kernel for GitHub Releases
├── src/
│   ├── main.c               # CLI entry point, argument parsing, dispatch
│   ├── runtime.h            # C-style ObjC runtime macros (no bracket syntax)
│   ├── vm.h / vm.m          # VZVirtualMachine lifecycle (create, start, stop)
│   ├── daemon.h / daemon.m  # Background daemon, Unix socket IPC, exec/stop
│   ├── console.h / console.m# PTY creation, raw terminal mode, I/O bridging
│   ├── setup.h / setup.m    # First-time download of kernel + arch.img
│   ├── state.h / state.m    # ~/.msl/ state file management
│   └── ipc.h / ipc.m        # Unix socket helpers for daemon IPC
└── plan.txt                 # This file
```

## Architecture

```
┌─────────────┐     Unix socket      ┌──────────────────┐
│  msl start   │─────────────────────►│  msl daemon      │
│  msl exec    │  ~/.msl/daemon.sock  │  (background)    │
│  msl stop    │◄─────────────────────┤                  │
└─────────────┘                      │  VZVirtualMachine │
                                     │  PTY (master fd) │
                                     │  stdin←→socket   │
                                     └────────┬─────────┘
                                              │
                                     ┌────────▼─────────┐
                                     │  Console/serial   │
                                     │  hvc0 getty@hvc0  │
                                     │  Arch Linux ARM   │
                                     │  ext4 (arch.img)  │
                                     │  VirtioFS /Users  │
                                     └──────────────────┘
```

## Key Decisions

| Aspect | Choice |
|---|---|
| Language | Objective-C in .m files, C-style via runtime macros |
| Build system | CMake, macOS 14.0+ target |
| Binary name | `msl` |
| VM resources | 2 vCPU, 2GB RAM |
| Directory sharing | `/Users` → `/Users` (transparent, no path translation) |
| Disk image | Pre-built arch.img on GitHub Releases, downloaded on first start |
| Kernel | Separate kernel file on GitHub Releases |
| `msl exec` | Serial console injection, `__MSL_EXIT:N` marker parsing |
| Lifecycle | ACPI shutdown, boot from scratch (no suspend) |
| Daemon | Background process holds VM + PTY, listens on Unix socket |
| Console attach | Ctrl+P Ctrl+Q to detach (VM keeps running) |
| State | `~/.msl/daemon.pid` + `~/.msl/daemon.sock` |

## CLI Commands

| Command | Action |
|---|---|
| `msl start` | Setup (download kernel+disk if missing), launch daemon, attach console |
| `msl stop` | Send ACPI shutdown via daemon, wait for VM to stop |
| `msl status` | Check daemon PID, report running/stopped |
| `msl exec <cmd>` | Send command to daemon via socket, stream output, return exit code |
| `msl version` | Print `msl 1.0.0` |

## IPC Protocol (Unix socket, ~/.msl/daemon.sock)

Byte 0 = command type:
- `0x01` CMD_CONSOLE_ATTACH — bidirectional console I/O forwarding
- `0x02` CMD_CONSOLE_DETACH — (unused, connection close = detach)
- `0x03` CMD_EXEC — send command, receive output + exit code
- `0x04` CMD_EXEC_RESULT — response to exec (exit_code + output_len + output)
- `0x05` CMD_STOP — stop daemon and VM
- `0x06` CMD_RESIZE — terminal resize (5 bytes: cmd + row + col + pad + pad)

Exec protocol:
- Client sends: CMD_EXEC (1B) + cmd_len (4B, network order) + cmd_data
- Daemon sends: CMD_EXEC_RESULT (1B) + exit_code (4B) + output_len (4B) + output

## VM Configuration

- Boot loader: VZLinuxBootLoader (kernel from ~/.msl/kernel)
- Kernel cmdline: `console=hvc0 root=/dev/vda rw loglevel=3`
- Storage: VZVirtioBlockDeviceConfiguration (arch.img via VZDiskImageStorageDeviceAttachment)
- Network: VZVirtioNetworkDeviceConfiguration + VZNATNetworkDeviceAttachment
- Directory sharing: VZVirtioFileSystemDeviceConfiguration ("MacShare") → VZDirectorySharingConfiguration + VZSharedDirectory -> /Users
- Serial: VZSerialPortConfiguration + VZFileHandleSerialPortAttachment (PTY slave)

## First-Time Setup Flow

1. `msl start` checks `~/.msl/kernel` and `~/.msl/arch.img`
2. If missing, downloads from GitHub Releases:
   - `https://github.com/<user>/msl/releases/download/v1.0.0/kernel`
   - `https://github.com/<user>/msl/releases/download/v1.0.0/arch.img`
3. Writes to `~/.msl/`

## Daemon Lifecycle

1. `daemon_launch()` forks a child process
2. Child: `setsid()` → closes stdio → `vm_start()` → `state_set_running(pid)` → `ipc_listen()` → accept loop
3. Accept loop handles: console attach (bidirectional PTY↔socket bridge), exec commands, stop
4. Parent: waits for state to show running, then `ipc_connect()` → sends CMD_CONSOLE_ATTACH → terminal I/O bridge

## Required Guest Configuration (arch.img)

The pre-built disk image must have:
- Arch Linux ARM root filesystem (ext4)
- `getty@hvc0.service` enabled (serial console login)
- `/etc/fstab`: `MacShare /Users virtiofs rw,noatime 0 0`
- Network DHCP on `eth0`
- Root user accessible
- Kernel at `/boot/Image` (copied to `kernel` during image build)

## Build & Sign

```cmake
find_library(VIRT_FW Virtualization REQUIRED)
find_library(FOUNDATION_FW Foundation REQUIRED)

target_link_libraries(msl PRIVATE ${VIRT_FW} ${FOUNDATION_FW})

add_custom_command(TARGET msl POST_BUILD
    COMMAND codesign --entitlements Resources/msl.entitlements -f -s - "$<TARGET_FILE:msl>"
)
```

## Build Image Script (scripts/build-image.sh)

Creates arch.img and extracts kernel for GitHub Release upload.
Run on Linux (or Docker with e2fsprogs). Steps:
1. `dd` a 4GB raw image
2. `mkfs.ext4 -L msl-root arch.img`
3. Download + extract Arch Linux ARM rootfs tarball
4. Configure: getty@hvc0, fstab, root password, network
5. Copy `/boot/Image` → `kernel`
6. Upload both to GitHub Releases

### Article 1: Original Specification

Build a lightweight CLI application called linux for macOS using Apple's Virtualization.framework.

The goal is to make Arch Linux feel like a native shell inside Terminal.app.

There is no GUI, no VM application, and no virtualization UI. The user should never have to think about virtual machines.

Primary command:

$ linux start

This should:
- Start (or resume) a lightweight Arch Linux ARM virtual machine.
- Attach the VM's serial console directly to the current terminal.
- Automatically change into the same working directory as the host, provided it is inside a shared mount.
- Drop the user directly into an interactive shell.

Example:

macOS:

~/Projects/myapp
$ pwd
/Users/felix/Projects/myapp

$ linux start

Arch Linux

felix@arch ~/Projects/myapp $

The current working directory should transparently map into the guest, e.g.

Host:
    /Users/felix

Guest:
    /Users/felix

or

Host:
    /Users/felix

Guest:
    /mnt/mac/Users/felix

The mapping should happen automatically before the shell starts.

Implementation:

- Swift
- Apple's Virtualization.framework
- Arch Linux ARM
- ARM64 kernel
- Virtio block device
- Virtio filesystem sharing
- Virtio networking
- Serial console attached to stdin/stdout
- Persistent virtual disk
- Suspend instead of power off whenever possible
- Resume existing VM if one already exists
- No graphical framebuffer
- No login manager
- Boot directly into a shell

CLI:

linux start      # resume or boot, then attach
linux stop       # shutdown guest
linux suspend    # suspend guest
linux status     # running/suspended/stopped

Goals:

- Feels like WSL.
- Startup should be nearly instant after the first launch.
- Zero GUI.
- Minimal memory overhead.
- Clean, readable Swift code.
- Fast enough that opening Linux becomes part of a normal terminal workflow rather than launching a virtual machine.

One extra feature I'd add is seamless command execution from macOS:

linux exec make
linux exec pacman -S neovim
linux exec cargo build

Internally it would reuse the already-running VM, execute the command in the corresponding mapped working directory, stream stdout/stderr back to the macOS terminal, and return the Linux process's exit code. That makes it feel much more like WSL, where Linux commands can be integrated naturally into your normal shell workflow.

1. Create the CLI

   * Build a Swift command-line tool called linux.
   * Parse commands (start, stop, status, suspend).
   * Store VM state under ~/.linux/.

2. Implement First-Time Setup

   * On first launch, download or prompt for an Arch Linux ARM root filesystem and kernel.
   * Create a persistent virtual disk.
   * Generate the initial VM configuration automatically.

3. Create the Virtual Machine

   * Use Apple's Virtualization.framework to configure:

     * ARM64 Linux VM
     * CPU and memory allocation
     * Virtio block device
     * Virtio network
     * Virtio filesystem sharing
     * Serial console

4. Boot Into a Shell

   * Boot directly into Arch Linux without any graphical output.
   * Connect the VM's serial console to the current terminal's stdin/stdout.
   * Ensure the user lands directly at a shell prompt.

5. Share the macOS Filesystem

   * Mount the user's home directory (or configurable directories) inside the VM using VirtioFS.
   * Preserve permissions where possible.
   * Make the shared mount available immediately after boot.

6. Mirror the Current Working Directory

   * Before attaching to the shell, determine the host's current working directory.
   * Translate it into the corresponding path inside the shared mount.
   * Automatically cd there so the Linux shell starts in the same project directory.

7. Implement VM Lifecycle

   * linux start resumes an existing VM or boots a new one.
   * linux suspend saves VM state.
   * linux stop performs a clean shutdown.
   * linux status reports whether the VM is running, suspended, or stopped.

8. Optimize Startup

   * Suspend the VM instead of shutting it down by default.
   * Reuse the existing VM between sessions.
   * Minimize boot time and resource usage to make launching Linux feel nearly instantaneous.

9. Polish the User Experience

   * Handle terminal resizing, Ctrl+C, Ctrl+D, and exit cleanly.
   * Display helpful errors if setup or boot fails.
   * Keep the interface simple so interacting with Linux feels like using another native shell rather than managing a virtual machine.

### Article 2: Homebrew Distribution Guide

Add Homebrew distribution support for the finished macOS Linux subsystem project.

Goal:

Users should be able to install the tool with:

    brew install msl

The binary should be available globally:

    msl start
    msl stop
    msl status

The project should be released as a proper Homebrew package.

---

Step 1: Prepare the release build

- Ensure the Swift project builds in release mode:

    swift build -c release

- Verify the generated binary works:

    .build/release/msl --version

- Ensure the binary does not depend on local development paths.
- Remove debug-only code.
- Add a version command.

Example:

    msl version

Output:

    msl 1.0.0
    macOS Subsystem for Linux

---

Step 2: Create a GitHub release

Create a public GitHub repository.

Repository example:

    github.com/<user>/msl

Create tagged releases:

    git tag v1.0.0
    git push origin v1.0.0

The release should contain:

    msl-1.0.0.tar.gz

containing:

    msl source code
    Package.swift
    LICENSE
    README.md

The release archive must have a stable URL.

Example:

    https://github.com/<user>/msl/archive/refs/tags/v1.0.0.tar.gz

---

Step 3: Create the Homebrew formula

Create a separate tap repository:

    homebrew-msl

Structure:

    homebrew-msl/
    └── Formula/
        └── msl.rb


The formula should:

- Download the GitHub release archive.
- Build the Swift executable.
- Install the binary into Homebrew's bin directory.

Example formula:

ruby
class Msl < Formula
  desc "macOS subsystem for Linux using Apple's Virtualization framework"
  homepage "https://github.com/<user>/msl"
  url "https://github.com/<user>/msl/archive/refs/tags/v1.0.0.tar.gz"
  sha256 "<SHA256>"
  license "MIT"

  depends_on :macos
  depends_on xcode: ["15.0", :build]

  def install
    system "swift", "build",
      "-c", "release",
      "--disable-sandbox"

    bin.install ".build/release/msl"
  end

  test do
    system "#{bin}/msl", "--version"
  end
end

### Article 3: Distribution Note

One important note: because this uses Apple's Virtualization.framework, the Homebrew formula should probably not ship the Arch image/kernel inside the package. The formula should only install the msl executable. The first msl start should download and initialize the Arch environment into something like:

~/.msl/
├── kernel
├── arch.img
├── config.json
└── snapshots/

That keeps brew install msl lightweight and follows Homebrew conventions.
